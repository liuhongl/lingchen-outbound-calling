Mac 软电话接入 9199 本地测试指导

本文档用于指导同事在 macOS 上替代 MicroSIP，使用 Mac 可用的 SIP 软电话接入本地 FreeSWITCH，并验证 sip-realtime-voice-gateway 的 9199 实时语音链路。

更新时间：2026-05-10

结论

Mac 适配是可行的，核心网关代码不需要因为替换 MicroSIP 而改动。

从第一性原理看，本地软电话只承担一件事：作为一个标准 SIP endpoint，与 FreeSWITCH 完成 SIP 注册、呼叫建立、RTP 音频收发和 codec 协商。只要 Mac 工具满足这些协议行为，FreeSWITCH 到网关的媒体链路仍然保持：

Mac SIP 软电话
  -> FreeSWITCH
  -> sip-realtime-voice-gateway
  -> 实时语音模型
  -> sip-realtime-voice-gateway
  -> FreeSWITCH
  -> Mac SIP 软电话

因此，替换点在 FreeSWITCH 前面的软电话工具，不在网关主逻辑。

适配 Mac 的过程中必须保证原有 Windows / MicroSIP 测试链路仍然可用。Mac 接入只是在本地软电话入口层新增一种客户端选择，不能通过改坏 Windows 注册、拨入 9199、控制面外呼或 FreeSWITCH 到网关媒体链路来换取 Mac 侧通过。

推荐工具

优先级如下：

  1. Linphone Desktop
  • 推荐作为首选 Mac 图形界面软电话。
  • 官方下载页显示支持 macOS、Windows、Linux 桌面客户端。
  • 下载地址：https://www.linphone.org/en/download/
  1. Zoiper
  • 推荐作为备选 Mac 图形界面软电话。
  • 官方页面显示支持 macOS、Linux、Windows、iOS、Android 和浏览器等平台。
  • 下载地址：https://www.zoiper.com/
  1. baresip
  • 推荐作为命令行/自动化验证备选。
  • 可通过 Homebrew 安装。
  • Homebrew 页面：https://formulae.brew.sh/formula/baresip

MicroSIP 官方定位是 Windows SIP softphone，不适合作为 Mac 同事的工具要求。官方主页：https://www.microsip.org/

当前本地测试参数

默认本地分机：

SIP user / extension: 1000
Password: tenlocal1000
SIP server: <运行 FreeSWITCH Docker 的宿主机局域网 IP>
SIP port: 5060
Transport: UDP
Dial number: 9199
Preferred codec: PCMA / G.711 A-law
Sample rate: 8000 Hz
Frame duration: 20 ms

如果多人同时测试，不建议共用 1000。可以改用 1001、1002 等 FreeSWITCH 已存在的测试分机，密码仍使用 tenlocal1000。控制面外呼时把 destination 改为对应分机号即可。

需要修改的地方

1. FreeSWITCH 本地地址

打开：

freeswitch-local/conf/vars.xml

确认以下三项指向运行 FreeSWITCH Docker 的宿主机局域网 IP，而不是旧的历史 IP：

<X-PRE-PROCESS cmd="set" data="domain=<HOST_LAN_IP>"/>
<X-PRE-PROCESS cmd="set" data="external_rtp_ip=<HOST_LAN_IP>"/>
<X-PRE-PROCESS cmd="set" data="external_sip_ip=<HOST_LAN_IP>"/>

例如宿主机 IP 是 192.168.0.165：

<X-PRE-PROCESS cmd="set" data="domain=192.168.0.165"/>
<X-PRE-PROCESS cmd="set" data="external_rtp_ip=192.168.0.165"/>
<X-PRE-PROCESS cmd="set" data="external_sip_ip=192.168.0.165"/>

修改后重启 FreeSWITCH 容器：

cd freeswitch-local
docker compose restart

2. Docker 端口

macOS Docker Desktop 本地软电话测试必须使用宿主机 RTP relay。当前可通结构是：

Mac 软电话 RTP 16384-16484
  -> rtp_host_relay.py
  -> 宿主机 26384-26484
  -> Docker 映射
  -> FreeSWITCH 容器 16384-16484

确认 docker-compose.yml 暴露了 SIP、ESL 和 relay 目标 RTP 端口：

ports:
  - "5060:5060/tcp"
  - "5060:5060/udp"
  - "5080:5080/tcp"
  - "5080:5080/udp"
  - "18021:8021/tcp"
  - "26384-26484:16384-16484/udp"

宿主机 relay 监听软电话看到的正常 RTP 端口：

python3 freeswitch-local/scripts/rtp_host_relay.py

如果本机已经有 TEN 的 relay 在后台运行，命令行通常类似：

rtp_host_relay.py --listen-start 16384 --listen-end 16484 --target-start 26384 --target-host 127.0.0.1

如果 Mac 和 FreeSWITCH 不在同一台机器，Windows/macOS 防火墙要允许：

5060/tcp
5060/udp
16384-16484/udp
26384-26484/udp

3. Codec 收敛

当前网关的电话媒体契约是：

PCMA / 8000 Hz / mono / 20 ms

Mac 软电话里应优先启用或只启用：

G.711 A-law / PCMA

可保留 G.711 u-law / PCMU 作为备选，但不要优先使用 Opus、G722、G729 等宽带或压缩 codec。否则 FreeSWITCH 可能发生转码，测试结果会混入额外变量。

如果希望 FreeSWITCH 侧也更确定，可以在 vars.xml 中把 codec 顺序收紧：

<X-PRE-PROCESS cmd="set" data="global_codec_prefs=PCMA,PCMU"/>
<X-PRE-PROCESS cmd="set" data="outbound_codec_prefs=PCMA,PCMU"/>

修改后同样需要重启 FreeSWITCH 容器。

4. 文案泛化

代码热路径不需要改。为了避免后续误解，可以把 README、测试页和诊断文案中的 MicroSIP 泛化成：

本地软电话（MicroSIP / Linphone / Zoiper）

这属于文档和 UI 文案改动，不是 Mac 适配的必要条件。

Linphone 配置建议

新建 SIP 账号时按以下思路填写：

Username: 1000
Display name: 1000
SIP domain / SIP server: <HOST_LAN_IP>
Password: tenlocal1000
Transport: UDP
Proxy: sip:<HOST_LAN_IP>:5060

音频 codec 设置：

Enable: PCMA / G.711 A-law
Optional: PCMU / G.711 u-law
Disable or lower priority: Opus, G722, G729

如果 Linphone 注册不上，优先检查：

1. Mac 和 FreeSWITCH 宿主机是否在同一局域网。
2. SIP server 是否填写宿主机 LAN IP，而不是 Docker 容器 IP。
3. FreeSWITCH 的 vars.xml 是否仍是旧 IP。
4. 防火墙是否拦截 5060/udp。

Zoiper 配置建议

新建 SIP 账号时按以下思路填写：

User / Login: 1000
Password: tenlocal1000
Domain / Host: <HOST_LAN_IP>
Port: 5060
Transport: UDP
Auth username: 1000

音频 codec 同样优先：

G.711 a-law / PCMA

如果 Zoiper 显示账号已注册但没有声音，通常不是账号问题，而是 RTP 方向问题。优先检查：

1. 16384-16484/udp 是否从 Docker 映射到宿主机。
2. 宿主机防火墙是否允许 UDP RTP。
3. vars.xml 的 external_rtp_ip 是否是宿主机 LAN IP。
4. Mac 是否开了 VPN 或代理导致路由绕行。

baresip 命令行备选

安装：

brew install baresip

配置账号时使用：

sip:1000@<HOST_LAN_IP>;auth_pass=tenlocal1000

baresip 更适合排查协议层问题。它没有图形界面软电话直观，但日志更适合判断注册、INVITE、SDP 和 RTP 是否正常。

验证步骤

A. 启动 FreeSWITCH

cd freeswitch-local
docker compose up -d --build

B. 启动网关

python -m app.main --config configs/local.example.toml

确认健康检查：

Invoke-RestMethod http://127.0.0.1:9100/health

C. Mac 软电话注册

在 Mac 软电话里注册 1000。注册成功后，可以在 FreeSWITCH 容器内检查：

docker exec -it sip_realtime_freeswitch fs_cli -x "show registrations"

期望看到类似：

1000,<HOST_LAN_IP>,...,sofia/internal/sip:1000@<MAC_IP>:<MAC_PORT>,...

D. 主动拨入验证

在 Mac 软电话拨：

9199

期望链路：

Mac softphone -> FreeSWITCH 9199 -> gateway media WebSocket -> 实时语音模型 -> Mac softphone

E. 控制面外呼验证

浏览器打开：

http://127.0.0.1:9100/outbound-test

或使用命令：

Invoke-RestMethod `
  -Method Post `
  -Uri http://127.0.0.1:9100/calls `
  -ContentType "application/json" `
  -Body (@{
    destination = "1000"
    external_call_id = "mac-softphone-test-001"
    caller_id_number = "9000"
  } | ConvertTo-Json)

期望流程：

网关 POST /calls
  -> FreeSWITCH Event Socket originate
  -> sofia_contact */1000 解析 Mac 当前注册 Contact
  -> 呼叫 Mac 软电话
  -> Mac 接听
  -> 进入 9199
  -> 接入实时语音模型

常见问题

1. 注册失败

优先检查：

账号：1000
密码：tenlocal1000
服务器：宿主机 LAN IP，不是 127.0.0.1，也不是 Docker 容器 IP
端口：5060
传输：UDP

如果 FreeSWITCH 的 vars.xml 仍指向旧 IP，也会导致注册或呼叫行为异常。

2. 能注册但拨 9199 不通

检查：

1. FreeSWITCH 容器是否运行。
2. 5060/udp 是否放行。
3. 9199 拨号计划是否仍存在。
4. 网关是否已经监听 9101 媒体端口。

9199 本地拨号计划应指向：

ws://host.docker.internal:9101/media/fs/${uuid}

3. 能接通但听不到声音

这通常是 RTP 或 codec 问题，不是网关 HTTP 控制面问题。

检查：

1. Mac 防火墙、宿主机防火墙、VPN。
2. rtp_host_relay.py 是否正在监听 16384-16484/udp。
3. Docker RTP 端口是否是 26384-26484 -> 16384-16484/udp。
4. vars.xml 的 external_rtp_ip 是否是宿主机 LAN IP。
5. 软电话 codec 是否使用 PCMA。

本次已确认的根因模式：

9197 能听到持续哔声：FreeSWITCH -> 软电话下行正常。
9196 没回声、9199 没声音：软电话 -> FreeSWITCH 上行 RTP 异常。
FreeSWITCH 统计：
  rtp_audio_in_packet_count = 0
  rtp_audio_in_media_bytes = 0
  rtp_audio_in_skip_packet_count 在涨

恢复 rtp_host_relay.py + 26384->16384 映射后：
  rtp_audio_in_packet_count > 0
  rtp_audio_in_media_bytes > 0

结论：当前独立项目不能裸用 Docker Desktop 直接暴露 RTP 给本机软电话；必须保留 host RTP relay 这层。

4. 外呼返回 USER_BUSY

含义通常是对端忙线或拒接。处理：

1. 确认 Mac 软电话没有占线。
2. 发起外呼后及时接听。
3. 不要同时在多个设备上共用同一个 1000 分机测试。
4. 主叫号码保持 9000，不要用 1000 呼叫 1000。

5. 外呼返回 NORMAL_TEMPORARY_FAILURE

常见原因是 Contact 瞬时不可达、NAT 或软电话注册状态陈旧。处理：

1. 重新注册软电话账号。
2. 重启 Mac 软电话。
3. 检查 show registrations 中的 Contact 是否是 Mac 当前 IP。
4. 检查 Mac 是否切换了网络、VPN 或热点。

验收标准

一次 Mac 软电话适配可以认为通过，需要满足：

1. Mac 软电话成功注册到 FreeSWITCH。
2. show registrations 能看到对应分机和 Mac 当前 Contact。
3. Mac 主动拨 9199 可以进入 AI 实时语音链路。
4. POST /calls destination=1000 可以呼叫到 Mac 软电话。
5. 接听后能进入 9199，并听到模型回复。
6. Windows / MicroSIP 原有链路仍然可以注册、拨入 9199，并通过控制面外呼接通。
7. 日志中没有持续的 codec、RTP、Event Socket 或 media WebSocket 错误。

不建议改的地方

Mac 软电话替换不应引发以下改动：

1. 不要重写网关 SIP/RTP 协议栈。
2. 不要绕过 FreeSWITCH 直接让软电话连网关。
3. 不要修改 Doubao S2S、Playout Engine、打断控制主逻辑。
4. 不要把 phone_codec 从 PCMA 改成 Opus/G722 来迎合某个软电话默认配置。

软电话只是本地测试入口。真实商用上线时，这个入口会继续被 SIP trunk 或企业中继替换，接通后的 9199 -> sip-realtime-voice-gateway -> 实时语音模型 主链路应保持稳定。
