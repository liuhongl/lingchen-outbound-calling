SIP 实时语音网关交接总文档

1. 当前结论

当前项目主线已经从旧 TEN 级联语音链路收敛为独立的 sip-realtime-voice-gateway 项目。

本项目目标是支撑 AI 外呼和电话实时对话：

业务系统
  -> sip-realtime-voice-gateway HTTP 控制面
  -> FreeSWITCH / SIP Trunk
  -> 电话用户
  -> FreeSWITCH 9199 媒体链路
  -> sip-realtime-voice-gateway 实时媒体网关
  -> 豆包 S2S 端到端实时语音模型
  -> 电话用户

旧 TEN 链路已经不作为当前商用主线。原因是旧链路是：

ASR final -> LLM -> TTS -> 电话播放

该结构串行等待多，体感延迟高。本项目当前主线使用端到端实时语音模型，电话音频持续进入模型，模型音频持续回到电话侧，并由网关负责播放时钟、打断和状态机。

2. 业务需求

当前模块面向 AI 外呼场景，业务系统需要能够：

1. 发起一通外呼。
2. 传入被叫号码和业务上下文。
3. 在电话接通后让 AI 与用户实时对话。
4. 支持用户插话打断旧回复。
5. 查询通话状态、接通耗时、通话时长和失败原因。
6. 后续支持真实 SIP trunk、业务回调、持久化和生产监控。

本地测试中使用软电话 1000 模拟真实被叫用户；Windows 可用 MicroSIP，macOS 可用 Linphone 或 Zoiper。真实商用时，软电话会替换成运营商 SIP trunk 或企业中继线路。

真实线路替换点在 FreeSWITCH 外呼 endpoint 层。接通后的 9199 -> 实时媒体网关 -> 豆包 S2S 链路应保持一致。当前实现仍使用 mod_audio_stream 作为 Media Adapter；商用阶段如果它在播放完成事件、打断停止、并发稳定性上压测不达标，再替换为定制 FreeSWITCH 模块或商用媒体适配层。

该链路在代码结构上可以按 call_id/session_id 多路复制，但当前还没有生产级并发上限、外呼队列和压测结论。本地 1000 软电话只能验证单路或小规模链路，不代表真实运营商并发能力。

4. 当前已经实现的能力

4.1 实时媒体主链路

已经实现：

1. FreeSWITCH 9199 把电话媒体接入网关 WebSocket。
2. 网关接收 8k / 20ms 电话上行 PCM。
3. 网关把电话音频重采样到模型输入格式。
4. 网关连接豆包 S2S 端到端实时语音模型。
5. 网关接收模型音频 delta。
6. 网关把模型音频重采样回 8k 电话帧。
7. 网关按电话媒体节奏回放到 FreeSWITCH。

当前主模型链路为豆包 S2S，不再保留阿里 Realtime、火山硬件智能体和旧 ASR/LLM/TTS 级联链路作为运行时回退。

4.2 播放和打断

已经实现：

1. Playout Engine：把模型输出转换成稳定电话下行帧。
2. Jitter / prefill：降低下行播放抖动。
3. FreeSWITCH playback 事件闭环：等待 chunk_played / queue_completed。
4. 用户插话时清空本地播放队列。
5. 用户插话时调用 uuid_audio_stream <uuid> break 停止 FreeSWITCH 侧旧音频。
6. 被打断的 assistant turn 不进入 committed history。
7. 通过尾部静音和播放完成确认避免最后几个字被吞或串入下一轮。

4.4 FreeSWITCH 外呼状态机

当前外呼状态机已经接入 FreeSWITCH channel 事件：

CHANNEL_PROGRESS / CHANNEL_PROGRESS_MEDIA -> ringing
CHANNEL_ANSWER                            -> answered
媒体 WebSocket 连接                        -> media_connected
CHANNEL_HANGUP / CHANNEL_HANGUP_COMPLETE  -> completed / busy / no_answer / failed

状态流转：

stateDiagram-v2
  [*] --> queued
  queued --> originating
  originating --> ringing: CHANNEL_PROGRESS
  originating --> answered: CHANNEL_ANSWER
  ringing --> answered: CHANNEL_ANSWER
  answered --> media_connected: media websocket connected
  media_connected --> completed: NORMAL_CLEARING
  answered --> completed: NORMAL_CLEARING
  originating --> busy: USER_BUSY / CALL_REJECTED
  ringing --> no_answer: NO_ANSWER
  originating --> failed: other hangup cause
  ringing --> failed: other hangup cause
  media_connected --> failed: other hangup cause

5.3 FreeSWITCH originate 命令

网关通过 Event Socket 执行：

originate {origination_uuid=<call_id>,origination_caller_id_name=AI_Assistant,origination_caller_id_number=9000,originate_timeout=30,hangup_after_bridge=true,sip_realtime_gateway_call_id=<call_id>}<endpoint> 9199 XML default

本地默认 endpoint：

sofia_contact:*/{destination}

因此 destination=1000 会先解析成当前软电话注册 Contact，例如：

sofia/internal/sip:1000@192.168.x.x:51611;ob;fs_nat=yes;fs_path=...

真实 SIP trunk 通常改为：

sofia/gateway/<trunk-name>/{destination}

5.4 电话媒体格式

电话侧核心媒体契约：

sample_rate = 8000 Hz
channels = 1
frame_duration_ms = 20
PCM frame = 160 samples * 2 bytes = 320 bytes
PCMA RTP payload = 160 bytes

含义：

8k         每秒 8000 个采样点，电话窄带语音常见格式。
20ms       每一帧代表 20ms 声音，是电话实时播放的基本节奏。
320 bytes  8k / 20ms / s16le / mono 的一帧 PCM 大小。

帧过大，会增加延迟；帧过小，会增加调度压力。当前 20ms 是电话系统常见折中。

5.5 模型媒体格式

当前网关内部会做采样率转换：

电话上行 8k PCM -> 模型输入 16k PCM
模型输出 24k PCM -> 电话下行 8k PCM

最终发回电话侧仍必须满足 8k / 20ms / 320 bytes。

8. 已知边界和未实现内容

当前未实现：

1. 通话记录持久化。当前 /calls 记录在内存中，网关重启后会清空。
2. 业务系统 webhook 回调。
3. 真实 SIP trunk 出局网关配置和运营商 CDR 对账。
4. 外呼任务队列、并发限流、失败重试、黑名单、时间窗。
5. 接口鉴权、签名、IP 白名单。
6. 生产指标面板和告警。
7. 业务上下文注入实时模型 system prompt。
8. 本地 FreeSWITCH IP 参数化。当前本地配置仍需要按测试机器 LAN IPv4 调整。

当前已知边界：

1. 本地软电话只验证 FreeSWITCH 之后的电话媒体链路，不能替代真实运营商测试。
2. 本地 NAT Contact 会变化，偶发 503 时先刷新软电话注册。
3. mod_audio_stream 必须按实时流使用，不能把模型音频攒成大块后再发。
4. 不要在同一通道上使用阻塞式 FreeSWITCH playback / speak 破坏实时媒体流。
5. 正常结束用 hangup_cause=NORMAL_CLEARING 表示，不应当作为 failure_reason。
6. 当前支持多通会话的代码结构，但尚未做生产级 max_active_calls、CPS、排队、重试和压测。
7. 真实 SIP trunk 的返回码、CDR、DTMF、NAT、防火墙和并发限制必须以运营商实测为准。

10. 后续实施路线

当前最大未知不是业务系统回调，也不是并发调度，而是真实运营商线路接入后 SIP 信令、PCMA/RTP、NAT、防火墙、挂机原因和媒体质量是否成立。因此下一阶段建议先做真实 SIP trunk 单路预验证，业务系统接入和并发护栏后移。

10.1 下一阶段：真实 SIP trunk 单路预验证

目标：

1. 配置真实 SIP trunk 出局网关。
2. 用测试页或 POST /calls 发起单通真实外呼。
3. 验证被叫号码格式、主叫号码规则、鉴权、PCMA、RTP、NAT、防火墙。
4. 验证接通后仍进入 9199，并复用当前实时媒体网关和豆包 S2S 链路。
5. 验证忙线、拒接、无人接、用户挂机、运营商错误码和 FreeSWITCH hangup_cause 映射。
6. 记录运营商 CDR / SIP trace 与网关状态字段的对应关系。

本阶段只做人工单路或极小规模验证，不做批量外呼。至少应限制测试号码，避免误拨、连拨和非预期外呼。

需要提前准备：

1. SIP trunk 服务器地址、端口、传输协议。
2. 鉴权方式：账号密码或 IP 鉴权。
3. 出局主叫号码规则。
4. 被叫号码格式：本地号、手机号、区号、86 或 +86。
5. RTP 端口范围、防火墙和 NAT 要求。
6. codec 要求：当前目标为 PCMA。
7. 运营商并发数、CPS、超时和风控限制。
8. 忙线、拒接、无人接、用户挂机等返回码说明。

10.2 后续阶段：业务系统接入基础能力

目标：

1. 外呼记录持久化，网关重启后仍能查询历史通话。
2. external_call_id 幂等，避免业务系统重复提交导致重复外呼。
3. completed / busy / no_answer / failed 等最终状态通过 webhook 回调业务系统。
4. webhook 增加 HMAC 签名、重试和失败记录。
5. context 中的业务参数可注入实时模型 system prompt。

10.3 后续阶段：并发与外呼调度护栏

目标：

1. max_active_calls：限制同时通话数。
2. max_originate_per_second：限制每秒发起数。
3. 外呼等待队列、超时回收和取消。
4. 失败重试、黑名单、时间窗、号码去重。
5. 并发压测，验证 FreeSWITCH、网关、豆包 S2S 和 SIP trunk 的共同容量。

10.4 后续阶段：商用媒体稳定性

目标：

1. 压测 mod_audio_stream 的 chunk_played / queue_completed 事件可靠性。
2. 压测 uuid_audio_stream break 的打断停止耗时和成功率。
3. 验证高并发下尾音完整、无旧回复串入、无持续卡顿。
4. 如果 mod_audio_stream 不达标，替换为定制 FreeSWITCH 模块或商用 Media Adapter。
